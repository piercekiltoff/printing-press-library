# feat: Ship explicit /pp-* slash commands for every Printing Press CLI skill

**Target repo:** printing-press-library

## Overview

Typing `/pp` in Claude Code returns no matches (see user screenshot 2026-04-20). The printing-press-library plugin only ships `plugin/skills/pp-*`; it has never shipped a `plugin/commands/` directory. Claude Code surfaces some plugin skills as slash commands automatically but does so inconsistently (compound-engineering's `ce:plan` appears as `/ce:plan`, but `pp-contact-goat` does not surface as `/pp-contact-goat` for this user even on a fresh plugin install).

The fix is to ship explicit `plugin/commands/pp-<slug>.md` shim files, one per CLI skill plus the `ppl` router, that invoke the corresponding skill with the user's arguments. This matches the pattern used by last30days-skill, codex-plugin-cc, vercel (Anthropic's official plugin), and others. Teach `tools/generate-skills/main.go` to emit commands alongside the skill mirrors so the two stay in sync as new CLIs are added.

## Problem Frame

Slash-command autocomplete is the primary discovery surface in Claude Code. The CLI ecosystem currently ships 22 skills and 1 router (`ppl`), all of which are functionally slash-invocable via `/<plugin>:<skill>` form (e.g. `/printing-press-library:pp-contact-goat`) but do not populate `/<name>` autocomplete matches for the user. The user's expectation, confirmed by screenshot and verbal ("pp not showing up wtf"), is that typing `/pp` should surface `/pp-contact-goat`, `/pp-slack`, `/pp-kalshi`, etc.

This is a discoverability regression from the user's perspective even though the repo never actually shipped commands. Given the pattern is load-bearing for onboarding ("how do I use this plugin?") and the fix is mechanical, it should land quickly.

## Requirements Trace

- R1. Typing `/pp` in Claude Code autocomplete MUST show `/pp-<slug>` entries for every CLI in `library/` that has a shipped SKILL.md.
- R2. Invoking `/pp-contact-goat <args>` MUST invoke the `pp-contact-goat` skill with the passed arguments, identical to how the skill is reached today via natural-language trigger or the fully-qualified `plugin:skill` form.
- R3. `/ppl` MUST be available as a slash command that invokes the `ppl` router skill.
- R4. The commands directory MUST be regenerated by `tools/generate-skills/main.go` whenever the generator runs, so new CLIs automatically gain a `/pp-<slug>` command without manual editing.
- R5. Command file frontmatter MUST carry the skill's description and argument-hint verbatim (so `/pp-contact-goat ?` shows the same help text as the skill).
- R6. Plugin version MUST be bumped (1.1.21 -> 1.1.22) and the marketplace index refreshed so users picking up the new version get the commands.
- R7. Existing skill-based invocation paths (natural language, `plugin:skill`) MUST keep working; commands are an additive surface, not a replacement.

## Scope Boundaries

- Not changing any SKILL.md content or CLI behavior. This is purely a discoverability/surface-area change.
- Not adding commands for any skill that does not have a SKILL.md on disk (a handful of newer CLIs ship without one; they stay skill-invocable but do not get a slash command until their SKILL.md is written).
- Not renaming or reorganizing existing skills.
- Not changing the `ppl` or `pp-<slug>` naming convention.

### Deferred to Separate Tasks

- Investigate why Claude Code's auto-surfacing of plugin skills as slash commands is inconsistent across plugins (ce:plan visible, pp-contact-goat not). Worth reporting to Anthropic but out of scope for this repo.
- Backfill SKILL.md for the CLIs that still don't have one (dominos-pp-cli, postman-explore, trigger-dev, pagliacci-pizza, steam-web, slack, linear per AGENTS.md). Those skills will gain slash commands automatically once their SKILL.md lands.

## Key Technical Decisions

- Decision: Command body is a single-line shim that tells Claude to invoke the named skill with `$ARGUMENTS`. Rationale: the skill already contains the full workflow prose; duplicating it in the command file would create drift. last30days's commands/last30days.md follows this exact pattern.
- Decision: Generate one command per CLI slug (`/pp-contact-goat`, `/pp-slack`, etc.) plus `/ppl` for the router. Rationale: matches existing skill naming; no cognitive remapping needed.
- Decision: `description` and `argument-hint` in the command file are copied verbatim from the corresponding SKILL.md frontmatter so `/pp-contact-goat ?` shows identical help to the skill-invocation surface. Rationale: single source of truth in library/**/SKILL.md; the command file is a derived artifact.
- Decision: `allowed-tools` is intentionally omitted in the command file (or kept loose) because the skill itself declares its own tool permissions. Rationale: the command is a skill-invocation shim; letting the skill's own declaration govern avoids divergence.
- Decision: Skip commands for skills that don't have a library SKILL.md (dominos-pp-cli, postman-explore, trigger-dev, pagliacci-pizza, steam-web, slack, linear per AGENTS.md). Rationale: without a description, a slash command is a dead end. Once those skills get SKILL.md (a separate workstream), regeneration auto-adds commands.
- Decision: Do not modify `plugin/.claude-plugin/plugin.json` beyond the version bump. Rationale: Claude Code discovers `commands/` by convention (see last30days example); no manifest key required.

## Open Questions

### Resolved During Planning

- Q: Does the plugin.json need a `commands: ./commands/` field? A: No. last30days-skill ships commands without any manifest key; discovery is conventional.
- Q: Should we also generate a top-level `/pp` alias that shows the catalog (like `/ppl`)? A: No, `/ppl` already fills this role. Adding a second alias just duplicates.
- Q: Should command files include an `allowed-tools` key? A: Leave it off. The skill's own declaration governs when the skill fires; requiring the command to re-declare creates drift.

### Deferred to Implementation

- Q: Exact filename of the new template file in `tools/generate-skills/`. Resolve during implementation based on what reads cleanest next to `skill-template.md`.
- Q: Whether to use a Go text/template or a simple string format for the command body. Resolve based on what the existing generator code uses for the skill side.

## System-Wide Impact

- **Interaction graph:** Every command file is a thin shim that invokes its skill. No new logic path; existing skill dispatch handles the rest. The `ppl` router skill is also reachable via `/ppl` in addition to its existing natural-language triggers.
- **Error propagation:** Unchanged. Errors surface from the skill's tool calls, not from the command shim.
- **State lifecycle risks:** None. Commands are stateless invocation shims; no cache, no persistence.
- **API surface parity:** This IS an API surface parity fix. It brings printing-press-library in line with last30days-skill, compound-engineering, codex-plugin-cc, and vercel, all of which ship explicit slash commands.
- **Integration coverage:** `.github/workflows/generate-skills.yml` already triggers on the relevant paths (registry.json, `library/**/.printing-press.json`, `tools/generate-skills/**`). Adding a new template file under `tools/generate-skills/` falls inside the existing trigger window.
- **Unchanged invariants:** No SKILL.md content changes. No CLI binary changes. No library/** Go source changes. Natural-language skill triggers keep working. The `plugin:skill` fully-qualified invocation form keeps working. Registry-only CLIs remain accessible via natural language / `ppl install` even though they do not gain a slash command.

## Risks & Dependencies

| Risk | Mitigation |
|------|------------|
| Claude Code plugin schema turns out to require a `commands` field in plugin.json that last30days and others are getting away without. | Test in Claude Code after regeneration; if autocomplete doesn't pick up the commands, add `"commands": "./commands/"` to plugin.json and re-bump. |
| Slash command name collisions with another installed plugin (e.g. another plugin also ships `/pp-slack`). | pp- prefix is specific enough that real-world collisions are unlikely. If one arises, rely on the `<plugin>:<command>` qualified form as the escape hatch. |
| Generator tests lock in template verbatim but the template evolves; tests become a burden. | Keep test assertions on shape (frontmatter keys present, description matches SKILL.md, body mentions skill name and $ARGUMENTS) rather than full-text comparisons. |
| Regeneration creates noise in the PR diff because of other stale artifacts. | Commit only the new commands + version bump + generator changes. Reject unrelated regen noise; if the generator rewrites skill mirrors identically, they should show no diff. |
